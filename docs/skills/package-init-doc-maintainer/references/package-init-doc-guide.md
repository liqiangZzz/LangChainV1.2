# 包级 __init__.py 文档维护细则

> 这是 `package-init-doc-maintainer` skill 的参考文档，服务对象是 Python 包的 `__init__.py`。
> 它不是整个项目文档体系，也不负责提前规划还没有代码承载的技术点。

## 使用场景

当用户要求维护、生成、检查 Python 包的 `__init__.py` 包说明时，先使用
`docs/skills/package-init-doc-maintainer/SKILL.md`，再按需参考本细则。

当前已落地的专项能力：维护项目中的 `__init__.py` 包说明。

- 让每个 Python 包的 `__init__.py` 成为“目录导航和学习说明”。
- 避免空文件、过时描述、和实际代码不一致的说明。
- 避免在 `__init__.py` 中导入示例模块，防止 import 包时触发真实 LLM 调用。

后续如果新增更多技术点，先补代码示例，再按实际目录和文件增量更新文档。
不要为了“看起来完整”提前写还不存在的模块、能力或学习路线。

## 入口关系

当前使用项目内 skill + reference 细则，不使用全局 skill。

原因：

- 这套规则强依赖本项目结构，例如 `models/`、`agents/`、
  `models/init_chat_model/init_chat_model_llm.py`、`.env` 约定。
- 项目内文档会跟随仓库一起变化，更适合记录当前已经落地的代码事实和命名历史。
- 规则只针对包级 `__init__.py` 说明，不应该命名成整个项目文档维护器。
- 不会影响你其他项目，避免把 LangChainV1.2 的局部约定污染成全局习惯。

什么时候适合升级成全局 skill：

- 多个项目都需要同一套文档维护方式。
- 流程稳定，不再频繁改。
- 需要复用脚本、模板、检查器，且不依赖某一个项目目录结构。
- 你希望以后在任意仓库里说一句“用 LangChain 文档维护 skill”，都能复用同一套能力。

简单判断：

- 只服务当前项目：用项目内 skill 和 reference 细则。
- 多个项目复用：再沉淀成 Codex 全局 skill。

## 总体原则

- 先读项目事实，再写说明，不能只按文件名猜。
- 只描述当前已经存在的代码、目录和示例，不提前补全“未来可能会写”的技术点。
- 文档要服务当前阶段学习：告诉读者“这个目录现在有什么、能学什么、运行有什么风险”。
- 示例说明和注释优先中文。
- 涉及真实模型调用、推理模型、联网、额度消耗时，必须明确提醒。
- 保持增量维护：新增知识点后再追加说明，不要把文档改成一篇泛泛的长教程。
- 如果发现某个知识点还没有代码示例，只能标记为“未覆盖/待补充”，不能写成已完成内容。

## 维护流程

1. 先运行扫描脚本：

   ```bash
   python scripts/audit_init_docs.py
   ```

2. 根据脚本输出确认：

   - 哪些目录是 Python 包。
   - 包下有哪些 `.py` 示例文件。
   - `__init__.py` 是否为空、是否已有文档字符串。
   - 描述是否和实际文件明显不一致。

3. 再读取需要调整的目录：

   - 先读该目录的 `__init__.py`。
   - 再读同目录下每个示例文件的开头、主要函数、主要调用方式。
   - 如果是新增子包，还要读并更新上一级 `__init__.py`，保证总览和子包说明一致。
   - 上级包只增加简短导航，详细文件说明仍放在子包自己的 `__init__.py` 中。

4. 修改 `__init__.py` 时只写包级说明，不导入示例模块。

   原因：本项目很多示例在模块顶层会调用真实 LLM。若在 `__init__.py` 中导入这些模块，
   可能导致 `import models` 或 `import agents` 时直接触发 API 调用。

5. 修改后执行最小验证：

   ```bash
   python -m py_compile <changed __init__.py>
   ```

   如果改了多个 `__init__.py`，可以一次性传入多个文件。

### 写作规范

- 默认使用中文。
- 描述要和当前文件真实内容一致，不要凭目录名编造能力。
- 优先写清楚“这个包演示什么”“主要文件是什么”“运行时需要注意什么”。
- 不要把 `__init__.py` 写成超长教程；详细代码解释应留在具体示例文件中。
- 涉及真实模型调用、推理模型、联网、额度消耗时，要明确提醒。
- 若目录名存在拼写历史问题，不要擅自重命名目录；可以在说明中注明实际含义。

### 推荐结构

```python
"""
<包名或主题>示例包。

本包用于演示……

主要文件：

- xxx.py
  演示……

- yyy.py
  演示……

运行注意事项：

- ……
"""
```

## 后续可扩展方向

这些方向仍然只服务于包级 `__init__.py` 说明，不代表当前项目已经完整覆盖：

- 已有示例索引维护
  在 `__init__.py` 中基于已经存在的代码生成目录索引，不提前规划不存在的章节。

- 专题说明维护
  在包级说明中只描述已经有示例文件支撑的专题。

- 注释规范维护
  仅作为判断包说明是否需要提醒的辅助信息，不直接批量改示例注释。

- 运行风险说明
  在包说明中标注哪些脚本会真实调用 LLM、哪些会调用推理模型、哪些可能多轮调用工具。

- 命名一致性检查
  发现目录或文件名拼写历史问题时，只记录和说明，不擅自重命名。
